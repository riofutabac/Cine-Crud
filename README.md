# Tema: Metodologías de Desarrollo de Software

## Introducción
- La metodología de desarrollo de software es crucial en la ingeniería de software.
- Se enfoca en aplicar métodos y procedimientos sistemáticos para alcanzar objetivos específicos.
- Patricio Michael Paccha Angamarca, experto en el tema, presenta el contenido.

## Tipos de Desarrollo de Software
- Grandes corporativos utilizan arquitecturas complejas y costosas.
- PyMES y start-ups prefieren arquitecturas simples y rentables.
- Proyectos académicos suelen no tener arquitectura y son de bajo costo.

## Definiciones Básicas
- Metodología: Estudio o elección de métodos aplicables a un objeto específico.
- Software: Conjunto de programas, reglas, documentación y datos asociados a un sistema de computación.
- Metodología de desarrollo de software: Conjunto integrado de técnicas y métodos para abordar cada actividad del ciclo de vida de un proyecto.

## Caracterización del Software
- El software es lógico e intangible, esencial para tareas específicas.
- Importancia de desarrollar proyectos metodológicamente debido a la complejidad y costo del software.
- Las metodologías son mejores prácticas que necesitan ser aplicadas.
- El factor humano es crucial en cualquier proyecto de software.

## Características del Software
- El software se desarrolla, no se fabrica como en otros campos.
- No se desgasta con el tiempo.
- La mayoría del software es a medida, no ensamblado de componentes existentes.
- Su base es puramente lógica.

## Metodologías Tradicionales
- Denominadas a veces como metodologías pesadas.
- Enfocadas en una documentación exhaustiva y control riguroso del proceso.


## Características de las Metodologías Tradicionales
- Enfocadas en la planificación completa antes de comenzar el desarrollo.
- Definición rigurosa de roles, actividades y artefactos.
- No se adaptan bien a cambios continuos; altos costos y falta de flexibilidad en entornos volátiles.

## Metodologías Ágiles
- Respuesta a los problemas de las metodologías tradicionales.
- Basadas en la planificación adaptativa y en retrasar decisiones.
- Incrementales, cooperativas, sencillas y adaptativas.
- Procesos con entregas frecuentes y comunicación constante entre clientes y desarrolladores.

## Principios de las Metodologías Ágiles
- Pautas y principios pragmáticos para una entrega menos complicada y más satisfactoria.
- Énfasis en la capacidad de respuesta a cambios sobre el seguimiento estricto de un plan.

## Metodologías Híbridas
- Combinan enfoques tradicionales y ágiles.
- Flexibilizan los procesos tradicionales sin perder su esencia.
- Orientadas a responder a necesidades del cliente y cambios en el mercado.

## Ventajas de las Metodologías Híbridas
- Adaptación a necesidades del proyecto y entornos dinámicos.
- Reducción de incertidumbre y minimización de riesgos mediante planificación y comunicación constante.
  
## Comparación entre Metodologías Tradicionales y Ágiles
- Tradicionales: Basadas en normas y estándares, control riguroso, roles y artefactos definidos.
- Ágiles: Basadas en heurísticas, adaptabilidad a cambios, menos control y documentación, cliente como parte del equipo.
- Tradicionales: Grupos grandes y distribuidos, énfasis en arquitectura, contratos rígidos.
- Ágiles: Grupos pequeños y locales, menor énfasis en arquitectura, contratos flexibles.

# Tema: Proceso de Desarrollo de Software

## Introducción al Proceso de Desarrollo de Software
- El desarrollo de software no es solo programar; implica múltiples actividades.
- Incluye ingeniería y análisis del sistema, análisis de requisitos, diseño, codificación, pruebas y mantenimiento.

## Clasificación de los Modelos de Software
- Modelos tradicionales: fases secuenciales sin considerar la naturaleza evolutiva del software.
- Modelos evolutivos: se adaptan a la evolución de los requisitos con el tiempo.
- Modelos orientados a objetos: alta iteratividad y solapamiento entre fases.
- Modelos basados en reutilización: enfatizan la reutilización sistemática del software.
- Procesos ágiles: desarrollo rápido y énfasis en la programación.
- Modelos para sistemas web: diseñados específicamente para aplicaciones web.

## Ejemplos de Modelos de Software
- Modelos tradicionales: clásico, lineal, en cascada, estructurado, basado en prototipos, RAD.
- Modelos evolutivos: incremental, iterativo, en espiral.
- Modelos para sistemas orientados a objetos: agrupamiento, Proceso Unificado.
- Modelos basados en reutilización: basado en componentes, Proceso Unificado.
- Procesos ágiles: programación extrema (XP), Scrum, desarrollo de software adaptativo, Crystal.
- Modelos para sistemas web: UML-based Web Engineering, modelos de Pressman.

## Modelo Primitivo
- Conocido como Modelo Prueba y Error o Modelo Codifica y Mejora.
- Inconvenientes: código pobremente estructurado, caro de desarrollar y mantener, posible rechazo del usuario por falta de análisis de requisitos.

## Modelos Lineales o Secuenciales
- Enfoque sistemático y secuencial, comenzando con análisis, diseño, codificación, pruebas y mantenimiento.
- Ofrecen control de progreso del proyecto pero no reflejan la realidad del desarrollo de software, que necesita iteraciones.

## Modelo Clásico o en Cascada
- Compuesto por fases secuenciales: análisis, diseño, codificación, pruebas y mantenimiento.
- Documentos de finalización de fase como criterio de avance.
- Definido en la norma estándar 2167-A del DoD de EEUU.
- Inconvenientes: rigidez, dificultad para manejar incertidumbre, ciclo largo, modelo monolítico.

## Modelo en V
- Variación del modelo en cascada.
- Relaciona actividades de prueba con análisis y desarrollo.
- Enfatiza desarrollo sincrónico de pruebas y desarrollo del programa.

## Modelos Basados en Prototipos
- Un prototipo es un modelo experimental con suficientes elementos para su uso.
- Objetivos: aclarar requisitos de usuarios, verificar viabilidad del diseño, probar suposiciones del sistema.
- Características: funcionamiento rápido, bajo costo de desarrollo, proceso iterativo.

## Desarrollo Rápido de Aplicaciones (RAD)
- Ciclo de desarrollo extremadamente corto, 60-90 días.
- Enfoque en integraciones constantes, más centrado en el código que en la documentación.
- Requiere compromiso y suficientes recursos humanos.

## Modelos Evolutivos
- Permiten desarrollar versiones cada vez más completas del software.
- Filosofía de poner un producto en explotación cuanto antes.
- Modelos evolutivos son iterativos y permiten entregas incrementales al cliente.

## Modelos Incrementales
- Sistema dividido en subsistemas funcionales.
- Cada nueva parte entregada es un incremento.
- Combina elementos del modelo en cascada y prototipos.

## Modelos Iterativos
- Entrega de un sistema completo desde el principio, cambiando funcionalidad en cada versión.
- Evolución de prototipos ejecutables y evaluables.
- Minicascada: cada iteración reproduce el ciclo de vida en cascada a menor escala.

## Modelos en Espiral
- Propuesto por B. Boehm, combina características del modelo clásico y prototipos.
- Enfatiza análisis de riesgo y avance desde el centro de la espiral hacia el exterior.
- Ventajas: realismo en el desarrollo de software, flexibilidad, manejo de riesgos y calidad.

## Modelos para Sistemas Orientados a Objetos
- Eliminación de fronteras entre fases, uso de componentes reutilizables.
- Desarrollo incremental y alto grado de iteratividad.
- Sistema se pone en producción gradualmente antes de estar completamente terminado.

## Proceso Unificado
- Definido por Rational Software Corporation, utiliza UML como lenguaje de modelado.
- Conducido por casos de uso, centrado en la arquitectura, iterativo e incremental.
- Cada ciclo consta de fases: inicio, elaboración, construcción y transición.

# Tema: Metodologías de Desarrollo de Software (Continuación)

## Introducción
- Una metodología describe cómo se organiza un proyecto de desarrollo de software.
- Incluye actividades/tareas con un enfoque sistémico, proporcionando herramientas concretas e intelectuales.
- Un enfoque sistemático representa la secuencia de pasos a realizar con sinergia.
- Metodologías cubren necesidades como: mejores aplicaciones, mejor proceso de desarrollo y establecer un proceso estándar en una organización.

## Metodología, Método y Ciclo de Vida del Software
- Una metodología puede seguir uno o varios modelos de ciclo de vida.
- El ciclo de vida indica qué obtener a lo largo del desarrollo, pero no cómo; esto lo define la metodología.
- La metodología es un concepto más amplio que el método.
- La metodología es un conjunto de métodos, y los métodos son un conjunto de técnicas.

## Metodología de Software
- Debe cubrir un proceso de ciclo de vida completo, abarcando aspectos de negocio y técnicos.
- Incluir un conjunto completo de conceptos y modelos consistentes, reglas y guías.
- Descripción completa de artefactos, notación soportada por herramientas CASE, técnicas probadas.
- Incluir métricas, asesoramiento sobre calidad, estándares y estrategias de prueba.
- Identificación de roles organizacionales y guías para la gestión de proyectos y aseguramiento de calidad.

## Metodologías Estructuradas
- Proponen una visión general del problema a un nivel de abstracción sencillo.
- Orientadas a procesos: creación de modelos del sistema que representan procesos, flujos y estructura de datos.
- Ejemplos: Merise, YSM, SSADM, METRICA.

## Metodologías Orientadas a Objetos
- Integran datos y procesos con un enfoque cercano al mundo real, facilitando la reutilización del software.
- Métodos dirigidos por datos: OMT, Fusion.
- Métodos dirigidos por responsabilidades: RDD, OBA.
- Métodos dirigidos por casos de uso: Objectory, Proceso Unificado.
- Métodos dirigidos por estados: Metodología de Shlaer y Mellor.

## Evolución de las Metodologías Orientadas a Objetos
- Metodologías de primera generación: Objectory.
- Metodologías de segunda generación: OMT, Syntropy, Fusion.
- Metodologías de tercera generación: Unificación y estandarización (UML), OPEN.

## Comparación entre Metodologías Estructuradas y Orientadas a Objetos
- Metodologías Estructuradas: Análisis, Diseño, Implementación (Relacional - Tablas).
- Metodologías Orientadas a Objetos: Análisis, Diseño, Implementación (Objetos).

## Paradigmas del Desarrollo: Metodologías Ágiles, Team Foundation Server y CMMI
- Visión tradicional: Metodologías pesadas (Métrica 3, Ciclo de Vida en Cascada, orientadas a proyectos grandes y complejos, mucha documentación).
- Nuevos paradigmas: Metodologías ligeras (CMMI, VS-TEAM 2012, MSF-Agile, MSF-CMMI, metodologías ágiles como XP y SCRUM, ciclo de vida iterativo incremental, adaptación al cambio, poca documentación).

## Procesos
- Los procesos son necesarios para mejorar.
- Mejora continua.

## CMMI
- Modelo de referencia para evaluar la madurez de los procesos de desarrollo de software de una organización.
- Proporciona guía para la mejora, es un modelo de buenas prácticas no prescriptivo.
- Se utiliza para: asesorar procesos internamente, evaluar la madurez de proveedores, detectar problemas, iniciar mejora, comparar con la industria.

## CMMI: Metodologías Ágiles y Pesadas
- Un modelo con distintas interpretaciones.
- Adaptarlo a las características de la organización.

## Herramienta Sí, Herramienta No
- Planificación, seguimiento, datos, estimación, pruebas, versiones, requisitos, solución técnica, cronograma, esfuerzos, incidencias, riesgos, RHH, gestión de cambios, captura de requisitos, cálculo, técnica, gestión, configuración, criterios, pruebas automáticas, planes, criterios de fin, diseño, programación, solución.

## Visual Studio: MSF for CMMI
- VS diseñado para gestionar procesos de ciclo de vida del software.
- Flexibilidad y control del proceso por la herramienta VS TEAM.
- "Plantillas de Metodologías" definen procesos soportados en VS, posibilidad de definir nuevas plantillas.
- Work Items representan tareas, bugs, escenarios, riesgos, requisitos.
- Creación y actualización desde Visual Studio, Excel, Project, portal web, herramientas de terceros.
- Control de las tareas.

## MSF for Agile Software Development
- MSF para desarrollo ágil: guía de procesos para el equipo de proyecto.
- Miembros del equipo asumen uno o más roles.
- Actividades agrupadas en workstreams.
- Productos de trabajo: documentos, hojas de cálculo, planes de proyecto, código fuente.
- Visual Studio Team System provee herramientas para implementar la guía de procesos.
- Uso de bases de datos de Work Items y Metric Warehouse para el seguimiento del proyecto.
- Control de versiones y portal de proyectos para almacenar productos de trabajo.

# Tema: Metodologías de Desarrollo de Software (Continuación)

## Metodologías Ágiles vs. Tradicional
- **Metodología Ágil**:
  - Pocos artefactos, modelado prescindible y desechable.
  - Roles genéricos y flexibles.
  - Contrato flexible, cliente parte del equipo de desarrollo in-situ.
  - Orientada a proyectos pequeños, corta duración, equipos pequeños y locales.
  - Arquitectura definida y mejorada a lo largo del proyecto.
  - Énfasis en aspectos humanos y trabajo en equipo.
  - Se esperan cambios durante el proyecto.
- **Metodología Tradicional**:
  - Más artefactos, modelado esencial y mantenido.
  - Roles específicos.
  - Contrato prefijado, cliente interactúa mediante reuniones.
  - Aplicable a proyectos de cualquier tamaño, especialmente grandes y dispersos.
  - Arquitectura definida tempranamente.
  - Énfasis en definición del proceso: roles, actividades y artefactos.
  - No se esperan cambios de gran impacto durante el proyecto.

# Tema: XP - Extreme Programming

## ¿Qué es XP?
- Extreme Programming (XP) es un proceso de software liviano con pocas reglas y prácticas.
- Basado en simplicidad, comunicación, feedback y coraje.
- Desarrollado por Kent Beck.
- Enfocado en adaptarse al cambio constante en software.
- Diseñado para equipos pequeños de desarrollo.

## Principios de XP
- **Simplicidad**: Desarrollar solo lo necesario, resolver necesidades actuales.
- **Feedback**: Desarrollos incrementales y entregas frecuentes, ajusta planes mediante retro-información.
- **Coraje**: Tomar decisiones difíciles, reparar errores rápidamente, mejorar el código continuamente.
- **Comunicación**: Comunicación continua y directa entre clientes y desarrolladores.

## Las 12 Prácticas de XP
1. **El Proceso de Planeación**: Definir valor de negocio, seleccionar aspectos a realizar.
2. **Pequeños Releases**: Sistema simple desde el inicio, actualizaciones frecuentes.
3. **Metáfora del Sistema**: Nombres y descripciones comunes, vocabulario compartido.
4. **Refactoring**: Mejorar diseño del sistema constantemente.
5. **Diseño Simple**: Construir el programa más simple que satisfaga los requisitos.
6. **Testing**: Desarrollar software escribiendo primero las pruebas.
7. **Pair Programming**: Dos programadores trabajando en la misma máquina.
8. **Propiedad Colectiva del Código**: Todos pueden cambiar cualquier parte del código.
9. **Integración Continua**: Integrar y construir el sistema múltiples veces al día.
10. **Cliente In-situ**: Cliente trabajando junto a los desarrolladores.
11. **Semana de 40 Horas**: No trabajar tiempo excesivo.
12. **Estándar de Codificación**: Escribir y documentar código de manera uniforme.

## Ciclo de Vida XP
- **Planeación**: Definir historias de usuario, estimar tiempos, crear plan de iteración.
- **Diseño**: Usar cartas CRC, crear soluciones prototipo, refactorización.
- **Codificación**: Escribir pruebas unitarias primero, programación en pares, integración continua.
- **Testing**: Todo el código debe tener y pasar pruebas unitarias, pruebas de aceptación.

## Roles en XP
- **Programador**: Escribir código y pruebas unitarias, definir tareas.
- **Cliente**: Escribir historias de usuario y pruebas funcionales, asignar prioridades.
- **Encargado de Pruebas**: Ayudar con pruebas funcionales, ejecutar pruebas regularmente.
- **Encargado de Seguimiento**: Verificar estimaciones, evaluar progreso, ajustar planes.
- **Entrenador (Coach)**: Responsable del proceso global, guiar al equipo en prácticas XP.
- **Gestor (Big Boss)**: Manejar recursos, resolver problemas, administrar reuniones.

## Puntos Claves de XP
- **Planeación**: Historias de usuario escritas por clientes, reunión de planeación para crear el plan de release.
- **Diseño**: Soluciones prototipo para reducir riesgo, refactorización constante.
- **Codificación**: Pruebas unitarias primero, programación en pares, integración secuencial.
- **Testing**: Pruebas de aceptación por clientes, pruebas de regresión frecuentes.

## Clima de Trabajo en XP
- **Espacio Abierto**: Mesas centrales y cubículos en el exterior.
- **Reunión Diaria**: Todo el equipo, discutir problemas y soluciones, evitar discusiones largas.
- **Participación del Cliente**: Cliente siempre disponible, iteraciones, pequeños incrementos.

## Conclusiones
- XP proporciona agilidad y flexibilidad en la gestión de proyectos.
- Se centra en crear productos según los requisitos exactos del cliente.
- Metodología conocida por su adaptabilidad a cambios constantes.

## Diferencia entre Metodología XP y Ágil
- XP es una metodología ágil con enfoque en comunicación y reutilización de código.
- Agile tiene roles definidos y iteraciones repetidas, XP se adapta y mejora continuamente.
